# Architecture Decision Records (ADR)

## ADR-001: Выбор фреймворка — Django 5.2

### Статус
Принято

### Контекст
Задание допускает использование Flask или Django. Необходимы аутентификация, работа с базой данных и шаблонами.

### Решение
Использовать Django 5.2.

### Обоснование
- Встроенная аутентификация, хеширование паролей и система пользователей.
- Встроенный ORM с миграциями.
- Встроенная поддержка Jinja2 в качестве альтернативного шаблонизатора.
- Меньше ручной инфраструктуры по сравнению с Flask.
- Версия 5.2 выбрана для совместимости с Python 3.14.

### Последствия
- Приложение следует структуре Django: проект + приложение + модели + view + URLconf.
- Админка Django не используется, но `django.contrib.auth` остаётся обязательной зависимостью.

## ADR-002: Выбор шаблонизатора — Jinja2 через встроенный бэкенд Django

### Статус
Принято

### Контекст
Задание требует использование шаблонов Jinja. Django имеет собственный шаблонизатор, но также поддерживает Jinja2.

### Решение
Использовать встроенный бэкенд `django.template.backends.jinja2.Jinja2` и настроить собственное Jinja2-окружение.

### Обоснование
- Это официально поддерживаемая конфигурация Django.
- Позволяет использовать Jinja2 без сторонних библиотек типа `django-jinja`.
- Сохраняет весь стандартный цикл запрос-ответ Django.

### Последствия
- Шаблоны проекта размещаются в `templates/jinja2/`.
- Шаблоны приложения размещаются в `library/jinja2/library/`.
- Необходимо создать файл `config/jinja2.py`, который возвращает сконфигурированный `jinja2.Environment` с глобалами `url` и `static`.
- Используется только Jinja2-шаблонизатор; DjangoTemplates не требуется, поскольку админка не используется.

## ADR-003: Выбор базы данных — SQLite

### Статус
Принято

### Контекст
Задание требует использования минимум одной базы данных: SQLite или DuckDB. Первоначально рассматривался DuckDB, но было принято решение от него отказаться.

### Решение
Использовать SQLite как единственную базу данных.

### Обоснование
- SQLite полностью поддерживается Django ORM из коробки.
- Не требует отдельного сервера и настройки.
- Подходит для локальной разработки и учебного проекта.
- DuckDB не имеет полноценного Django-бэкенда для записи и миграций; единственный известный вариант (`django-duckdb-readonly`) read-only и не подходит для CRUD-приложения.

### Последствия
- Конфигурация БД: `django.db.backends.sqlite3`.
- Файл базы данных: `/app/data/db.sqlite3` внутри контейнера, в volume `library-data`.
- В локальной разработке без контейнера путь переопределяется переменной окружения `DB_PATH`; если она не задана и `/app/data` отсутствует, используется `db.sqlite3` в корне проекта.
- Используются стандартные миграции Django (`makemigrations`, `migrate`).

## ADR-004: Подход к аутентификации — django.contrib.auth

### Статус
Принято

### Контекст
Требуется регистрация, вход, выход, хеширование паролей и проверка существующих пользователей.

### Решение
Использовать стандартную модель `User` из `django.contrib.auth` и форму `UserCreationForm` с расширением для проверки уникальности email.

### Обоснование
- Django предоставляет всё необходимое из коробки.
- Пароли хешируются через PBKDF2 по умолчанию.
- Проверка уникальности `username` уже встроена; `email` проверяется дополнительно.
- Нет необходимости создавать собственную модель пользователя.

### Последствия
- Email является уникальным и обязательным для регистрации.
- Регистрация реализуется через собственную view и форму.
- Защищённые маршруты используют декоратор `@login_required`.
- При неудачном входе отображается `non_field_errors` формы.

## ADR-005: Модель данных — Book и Borrowing

### Статус
Принято

### Контекст
Необходимо хранить книги и историю их выдачи. Возможны два подхода: хранить текущего читателя в книге или вести полную историю выдач.

### Решение
Использовать две модели: `Book` и `Borrowing`.

### Обоснование
- `Book` содержит статические данные о книге и владельце.
- `Borrowing` содержит каждую выдачу книги, включая даты выдачи, возврата и срок возврата.
- Активная выдача определяется полем `returned_at=None`.
- История чтения сохраняется автоматически при возврате книги.
- Это позволяет легко отвечать на вопросы: кто сейчас держит книгу, кто раньше читал, сколько раз книга выдавалась.

### Последствия
- Для проверки доступности книги используется запрос к `Borrowing` с `returned_at__isnull=True`.
- Возврат книги означает установку `returned_at=now()` для активной записи.
- Нельзя взять книгу, если существует активная выдача.

## ADR-006: Ролевая модель — отсутствует

### Статус
Принято

### Контекст
Пользователь указал, что строгая система ролей не нужна, все пользователи — друзья.

### Решение
Не использовать группы, пермиссии или роли. Авторизация основывается на владении объектом.

### Обоснование
- Упрощает код и снижает объём работы.
- Соответствует требованию «мы друзья».
- Достаточно проверки `book.owner == request.user` для редактирования/удаления.

### Последствия
- Все авторизованные пользователи видят все книги.
- Любой авторизованный пользователь может взять свободную книгу.
- Право вернуть книгу есть у текущего читателя или владельца.

## ADR-007: Стили — собственный CSS в стиле editorial-минимализма

### Статус
Принято

### Контекст
Пользователь пожелал минималистичный и профессиональный интерфейс без ощущения "сгенерированности" и предоставил референсный сайт с чистым editorial-дизайном.

### Решение
- Отказаться от Pico.css и построить собственный CSS на основе референсного дизайна.
- Использовать шрифт Inter через Google Fonts.
- Применить монохромную палитру: белый фон `#FFFFFF`, чёрный текст `#1A1A1A`, серые вторичные элементы `#6B6B6B`, тонкие серые границы `#E5E5E5`.
- Использовать плоские кнопки и поля ввода одинаковой высоты (`44px`).
- Оформить header фиксированным, с блюром и тонкой нижней границей.
- Использовать карточки с тонкими границами вместо теней.
- Сделать бейджи монохромными.
- Убрать эмодзи из интерфейса.
- Footer идёт естественно после контента.

### Обоснование
- Референсный сайт демонстрирует цельный профессиональный визуальный язык.
- Отказ от эмодзи, теней и ярких цветов убирает ощущение "сгенерированности".
- Собственный CSS даёт полный контроль над высотой кнопок и полей, типографикой и отступами.

### Последствия
- Файл `static/css/style.css` полностью переписан и не зависит от Pico.css.
- В `base.html` подключается шрифт Inter.
- Все шаблоны используют новые CSS-классы.
- Каталог представлен сеткой карточек в новом визуальном стиле.

## ADR-008: Поиск и сортировка книг — серверная обработка

### Статус
Принято

### Контекст
Требуется поиск по названию и автору, а также сортировка по разным полям.

### Решение
Реализовать поиск и сортировку на стороне сервера через параметры query string.

### Обоснование
- Для учебного проекта объём данных невелик, серверная фильтрация достаточна.
- Проще реализовать и тестировать, чем клиентский JavaScript.
- Параметры query string сохраняют состояние при обновлении страницы и передаче ссылки.

### Последствия
- В `BookListView` обрабатываются параметры `q` (поиск), `sort` (поле сортировки) и `order` (направление).
- Используются ORM-методы `filter` и `order_by`.
- В шаблоне отображаются текущие значения фильтров.
- По умолчанию сортировка — по названию (`title`), по убыванию (`desc`).

## ADR-009: Ограничение срока возврата — month-picker + не более 365 дней

### Статус
Принято

### Контекст
Пользователь указал, что срок возврата задаётся при выдаче и не может превышать 365 дней. При этом точный день не критичен, достаточно выбора месяца и года. Важно, чтобы виджет давал понятный date picker и подсказку формата.

### Решение
- Использовать `DateField` с `input_formats=["%Y-%m"]` и виджетом `DateInput(attrs={"type": "month", "placeholder": "2026-08"})`.
- В шаблоне формы выдачи добавить текстовую подсказку: «Формат: 2026-08 (год-месяц).»
- При получении значения в формате `YYYY-MM` в методе `clean_due_date` устанавливать день равным 1.
- Валидация 365 дней и запрета прошедших дат сохраняется.

### Обоснование
- Пользователю достаточно выбрать месяц и год возврата; точный день не критичен для домашней библиотеки.
- `DateField` с `input_formats` корректно обрабатывает `YYYY-MM` и позволяет Django рендерить нативный month picker.
- Placeholder и подсказка решают проблему непонятного формата ввода.
- Автоматическая подстановка 1-го числа делает форму более forgiving.

### Последствия
- При выдаче книги отображается нативный month picker (в современных браузерах) с placeholder и подсказкой.
- День возврата всегда равен 1-му числу выбранного месяца.
- Валидация продолжает ограничивать срок 365 днями.

## ADR-010: Язык интерфейса — русский

Принято

## ADR-011: Контейнеризация и деплоймент — Podman + Gunicorn + Nginx + GHCR

### Статус
Принято

### Контекст
Пользователь планирует развёртывать приложение на VPS с использованием контейнеров, Gunicorn в качестве WSGI-сервера, Nginx в качестве reverse proxy, а GitHub Container Registry для хранения образов.

### Решение
- Собирать production-образ с помощью `Containerfile` на основе `python:3.14-slim`.
- Использовать Gunicorn как WSGI-сервер.
- Использовать Nginx для проксирования запросов к Gunicorn и раздачи статических файлов.
- Хранить SQLite в именованном volume `library-data`, статику — в именованном volume `library-static`.
- Настроить GitHub Actions для автоматической сборки и публикации образа в GHCR при пуше в `main`.
- Конфигурировать Django через переменные окружения: `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`, `CSRF_TRUSTED_ORIGINS`.

### Обоснование
- Gunicorn надёжнее и безопаснее встроенного `runserver`.
- Nginx эффективно раздаёт статику и изолирует Gunicorn от прямого доступа из интернета.
- Volume для SQLite обеспечивает сохранность данных при обновлении образа.
- Volume для статики позволяет nginx получать актуальные файлы после `collectstatic`.
- GHCR интегрирован с GitHub и не требует отдельного registry.
- Переменные окружения позволяют использовать один образ для разных сред.

### Последствия
- Добавлены файлы `Containerfile`, `entrypoint.sh`, `podman-compose.yml`, `nginx.conf`, `.env.example`, `.dockerignore`, `.github/workflows/build-and-push.yml`.
- `config/settings.py` читает production-настройки из `os.environ`.
- SQLite находится в `/app/data/db.sqlite3` внутри контейнера.
- Статика собирается в `/app/staticfiles` через `python manage.py collectstatic`.
- Деплой на VPS выполняется вручную: `podman-compose pull && podman-compose up -d`.
- TLS не обрабатывается внутри приложения; upstream load balancer терминирует HTTPS.

## ADR-012: Сортировка каталога по умолчанию — по названию, по убыванию

### Статус
Принято

### Контекст
Пользователь указал, что при открытии каталога книги должны сортироваться по названию.

### Решение
- В `BookListView.get_queryset()` использовать `sort = self.request.GET.get("sort", "title")`.
- В `BookListView.get_context_data()` передавать `sort = "title"` по умолчанию.
- В шаблоне `book_list.html` сделать опцию «По названию» выбранной по умолчанию.
- Направление сортировки по умолчанию — убывание (`desc`).

- Обоснование: пользователь явно запросил сортировку по названию по умолчанию; убывание соответствует обратному алфавитному порядку (Я → А).
- Сохраняется возможность переключить сортировку и направление через интерфейс.

### Последствия
- При открытии `/books/` книги отображаются, отсортированные по названию от Я до А.
- Query string сохраняет выбор пользователя.

## ADR-013: Обработка ошибок входа — отображение non_field_errors

### Статус
Принято

### Контекст
При неудачном входе Django не добавляет ошибку к конкретному полю, а возвращает `non_field_errors`. В изначальном шаблоне эти ошибки не отображались.

### Решение
- В `templates/jinja2/login.html` добавить блок вывода `form.non_field_errors()` в виде заметного сообщения.
- Сохранить вывод ошибок отдельных полей.

### Обоснование
- Пользователь должен видеть, почему вход не удался.
- `non_field_errors` — стандартный механизм Django для ошибок, не привязанных к полю.

### Последствия
- При неверном логине или пароле отображается сообщение «Пожалуйста, введите правильные имя пользователя и пароль.»
- Шаблон login.html содержит отдельный блок для non_field_errors.
